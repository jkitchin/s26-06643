# Software Forge Academy Characters

Meet the cast of characters who will guide you through scientific software development. Each character embodies a key tool, practice, or mindset that will help you become a skilled software craftsperson.

---

## The Tool Masters

Characters who embody essential development tools you'll use throughout the course.

### Gina Git 🔀
*The Version Keeper*

**Concept:** Version control and git workflows

**Personality:** Never forgets anything—she has perfect memory of every change ever made. She thinks in branches and loves a clean merge. Gets anxious when she sees uncommitted changes.

**Catchphrase:** *"Commit early, commit often!"*

**Appears in:** Git Version Control (02), GitHub Collaboration (08)

**Teaches:**
- Branching and merging strategies
- Commit message best practices
- Resolving conflicts gracefully
- Git workflows (feature branches, PRs)

**Study tip:** *"A good commit message explains why, not just what. Future you will thank present you."*

---

### Terry Terminal 💻
*The Command Line Companion*

**Concept:** Terminal and shell proficiency

**Personality:** Old-school but efficient. Believes the keyboard is mightier than the mouse. Speaks in short, precise commands. Loves pipes and redirects.

**Catchphrase:** *"Why click when you can type?"*

**Appears in:** Terminals (01), throughout course

**Teaches:**
- Shell navigation and commands
- Environment variables
- SSH and remote connections
- Shell scripting basics

**Study tip:** *"Learn the shortcuts. Your future self will type thousands fewer keystrokes."*

---

### Tessa Test ✅
*The Quality Guardian*

**Concept:** Testing and test-driven development

**Personality:** Paranoid in the best way—she never trusts code until it's proven. Thorough and systematic. Celebrates when tests pass, investigates relentlessly when they fail.

**Catchphrase:** *"If it's not tested, it's broken."*

**Appears in:** Testing with AI (05), Code Quality (06)

**Teaches:**
- Unit testing with pytest
- Test-driven development (TDD)
- Mocking and fixtures
- Code coverage analysis

**Study tip:** *"Write the test first. It clarifies what you're actually trying to build."*

---

### Doc Douglas 📝
*The Documentation Champion*

**Concept:** Documentation and code communication

**Personality:** Communicative and clear. Believes code without documentation is a ticking time bomb. Writes for his future self and his colleagues.

**Catchphrase:** *"Future you will thank present you."*

**Appears in:** Documentation (10), throughout course

**Teaches:**
- Docstrings and type hints
- README writing
- Sphinx documentation
- API documentation

**Study tip:** *"Write your docstring before you write your function. It forces you to think about what it should do."*

---

### Vinny Virtual 📦
*The Environment Isolator*

**Concept:** Virtual environments and dependency management

**Personality:** Isolated and clean. Believes every project deserves its own space. Gets visibly upset when someone installs packages globally.

**Catchphrase:** *"Keep your dependencies to yourself!"*

**Appears in:** Python Packaging (04), throughout course

**Teaches:**
- Virtual environments (venv, conda)
- requirements.txt and pyproject.toml
- Dependency resolution
- Reproducible environments

**Study tip:** *"Always activate your environment first. Always."*

## The Practice Guides

Characters who embody essential software development practices.

### Deb Debugger 🔍
*The Bug Detective*

**Concept:** Debugging and troubleshooting

**Personality:** Patient and methodical. Approaches every bug as a mystery to solve. Never guesses—always investigates systematically.

**Catchphrase:** *"The bug is always where you least expect it."*

**Appears in:** All lectures, especially Testing (05), Code Quality (06)

**Teaches:**
- Reading error messages
- Using debuggers (pdb, IDE debuggers)
- Print debugging done right
- Rubber duck debugging

**Study tip:** *"Read the error message. All of it. The answer is usually there."*

---

### Ricky Refactor ♻️
*The Code Improver*

**Concept:** Refactoring and code improvement

**Personality:** Never satisfied with "good enough." Iterative thinker who believes code can always be cleaner. Knows when to stop, though.

**Catchphrase:** *"Make it work, make it right, make it fast—in that order."*

**Appears in:** Code Quality (06), Python Classes (07)

**Teaches:**
- Code smells and how to fix them
- Extract function/method patterns
- DRY (Don't Repeat Yourself)
- When to refactor vs. when to ship

**Study tip:** *"Refactor in small steps. Run tests after each change."*

---

### Paula Pattern 🏗️
*The Architecture Thinker*

**Concept:** Design patterns and software architecture

**Personality:** Experienced and thoughtful. Sees the big picture. Knows that most problems have been solved before—you just need to recognize the pattern.

**Catchphrase:** *"Don't reinvent the wheel—use a pattern."*

**Appears in:** Python Classes (07), APIs (03)

**Teaches:**
- Common design patterns
- When to use classes vs. functions
- API design principles
- Modular architecture

**Study tip:** *"Before you code, sketch the structure. Five minutes of design saves five hours of debugging."*

---

### Percy Performance ⚡
*The Speed Optimizer*

**Concept:** Performance and optimization

**Personality:** Speed-obsessed but measured. Never optimizes without profiling first. Knows that premature optimization is the root of all evil.

**Catchphrase:** *"Profile first, optimize second."*

**Appears in:** Code Quality (06), Testing (05)

**Teaches:**
- Profiling code
- Big-O thinking
- When optimization matters
- Caching strategies

**Study tip:** *"Don't guess what's slow. Measure it."*

---

## The AI Collaborators

Characters representing AI-assisted development.

### Claude Coder 🤖
*The AI Pair Programmer*

**Concept:** AI-assisted development with Claude

**Personality:** Helpful and knowledgeable, but works best with clear instructions. Admits when uncertain. Great at explaining and suggesting, but you're the decision-maker.

**Catchphrase:** *"Let me help you think through this."*

**Appears in:** AI-Assisted Development (00), Testing with AI (05), Advanced MCP (11)

**Teaches:**
- Effective prompting for code
- Code review with AI
- When to use AI vs. when to think yourself
- AI limitations and verification

**Study tip:** *"Be specific in your requests. 'Fix this bug' is less helpful than 'This function returns None when it should return an empty list.'"*

---

**Concept:** Model Context Protocol and tool integration

**Personality:** Connector and integrator. Loves making different tools work together. Gets excited about APIs and protocols.

**Catchphrase:** *"Let's connect that!"*

**Appears in:** Working with APIs (03), Advanced MCP (11)

**Teaches:**
- MCP server concepts
- Tool integration

## The Collaboration Crew

Characters who embody teamwork and collaboration practices.

### Review Rachel 👀
*The Code Reviewer*

**Concept:** Code review and constructive feedback

**Personality:** Constructive and detailed. Finds issues but always suggests solutions. Knows that code review is about making code better, not about being right.

**Catchphrase:** *"This works, but have you considered...?"*

**Teaches:**

- How to receive feedback gracefully

## The Study Squad

Characters who model good learning practices.

### Practice Pixel 🎮
*The Hands-On Learner*

**Concept:** Learning by doing

**Personality:** Energetic and hands-on. Learns by building, not just reading. Always has a side project going.

**Catchphrase:** *"Don't just read it—code it!"*

**Teaches:**
- Active learning strategies
- Building portfolio projects
- Learning from mistakes

**Study tip:** *"For every concept, write code that demonstrates it. That's when learning sticks."*

---

### Query Quinn 🤔
*The Question Asker*

**Concept:** Curiosity and deep understanding

**Personality:** Endlessly curious. Never satisfied with "it just works"—wants to know why. Her questions often reveal the deepest insights.

**Catchphrase:** *"But why does that work?"*

**Teaches:**
- Asking good questions
- Debugging through inquiry
- Stack Overflow etiquette

**Study tip:** *"If you can't explain why it works, you don't really understand it yet."*

---

### Rita Readable 📖
*The Clarity Champion*

**Concept:** Code readability and maintainability

**Personality:** Values clarity above all. Believes code is read more than written. Names things well and formats consistently.

**Catchphrase:** *"Code is for humans first, computers second."*

**Teaches:**
- Meaningful variable names
- Code formatting (Black, isort)
- Self-documenting code
- When comments help vs. hurt

**Study tip:** *"If you need a comment to explain what code does, maybe the code should be clearer."*

## Using Characters in the Course

Characters can appear in:

1. **Lecture introductions** - Character introduces the topic
2. **Quiz questions** - "What would Tessa Test say about this?"
3. **Assignment hints** - Characters offer guidance
4. **Trivia games** - Questions associated with character mascots
5. **Adventure story** - Characters as mentors in the Software Forge
6. **Error messages** - Friendly character-themed debugging help
